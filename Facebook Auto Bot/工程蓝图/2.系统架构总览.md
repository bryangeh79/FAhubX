# 2. 系统架构总览

## 架构设计原则

### 核心原则
1. **用户控制优先**：用户控制VPN/IP，降低中心化风险
2. **PWA优先**：移动端友好，支持离线功能
3. **静默运行**：无故障不打扰用户
4. **弹性设计**：预期失败，设计恢复机制

### 技术原则
1. **微服务架构**：模块化，独立部署
2. **事件驱动**：松耦合，高扩展性
3. **最终一致性**：允许短暂的数据不一致
4. **防御性编程**：假设外部系统不可靠

## 核心服务分解

### 1. 账号管理服务 (Account Service)
```yaml
职责:
  - Facebook账号的CRUD操作
  - 会话状态管理 (cookies, localStorage)
  - VPN/IP配置管理
  - 账号健康检查

技术栈:
  - 框架: NestJS
  - 数据库: PostgreSQL (账号表、会话表)
  - 缓存: Redis (会话缓存)
  - 通信: gRPC (内部服务调用)
```

### 2. 任务调度服务 (Scheduler Service)
```yaml
职责:
  - 秒级精度任务调度
  - 并发控制 (账号锁)
  - 任务队列管理
  - 失败重试策略

技术栈:
  - 框架: NestJS + Bull (队列)
  - 存储: Redis (队列、锁)
  - 调度: node-cron (秒级调度)
  - 监控: Prometheus + Grafana
```

### 3. 对话引擎服务 (Conversation Engine)
```yaml
职责:
  - 对话剧本管理 (50个剧本)
  - 对话协调执行
  - 自然时机调度
  - 内容个性化

技术栈:
  - 框架: NestJS
  - AI集成: OpenAI API (可选)
  - 存储: PostgreSQL (剧本库)
  - 缓存: Redis (对话状态)
```

### 4. 执行节点服务 (Execution Node)
```yaml
职责:
  - 本地浏览器自动化
  - VPN连接管理
  - 任务执行监控
  - 结果回传

技术栈:
  - 浏览器: Puppeteer + Stealth Plugin
  - VPN: OpenVPN/WireGuard客户端
  - 通信: WebSocket (实时状态)
  - 容器: Docker (环境隔离)
```

### 5. 通知服务 (Notification Service)
```yaml
职责:
  - 应用内通知管理
  - 推送通知发送
  - 通知偏好设置
  - 通知历史记录

技术栈:
  - 框架: NestJS
  - 推送: Firebase Cloud Messaging
  - 存储: PostgreSQL (通知记录)
  - 实时: Socket.io (WebSocket)
```

## 数据流设计

### 正常执行流程
```
1. 用户创建自动聊天配置
2. 配置保存到数据库，调度器注册任务
3. 调度时间到达，检查并发控制
4. 获取账号锁，分配执行节点
5. 执行节点启动浏览器，连接VPN
6. 执行对话剧本，实时状态回传
7. 执行完成，释放账号锁
8. 记录执行结果，静默成功
```

### 失败处理流程
```
1. 执行失败，捕获异常
2. 根据失败类型分类 (账号/网络/内容/系统)
3. 生成用户友好的错误消息
4. 发送应用内通知
5. 根据重试策略决定是否重试
6. 记录失败详情，开始7天倒计时
```

### 离线同步流程
```
1. 用户离线时操作
2. 操作存入本地IndexedDB
3. Service Worker标记为待同步
4. 网络恢复，后台同步
5. 冲突检测和解决
6. 同步完成，更新本地缓存
```

## 部署架构

### 开发环境
```yaml
部署方式: Docker Compose
组件:
  - 前端: React开发服务器
  - 后端: NestJS开发服务器
  - 数据库: PostgreSQL容器
  - 缓存: Redis容器
  - 消息队列: RabbitMQ容器
```

### 生产环境
```yaml
部署方式: Kubernetes集群
组件分布:
  - 前端: CDN + 静态托管
  - API网关: Kubernetes Ingress + Nginx
  - 业务服务: Kubernetes Deployment (多副本)
  - 数据库: 托管PostgreSQL (如AWS RDS)
  - 缓存: 托管Redis (如AWS ElastiCache)
  - 文件存储: S3兼容存储 (如MinIO)
```

### 执行节点部署
```yaml
部署选项:
  选项A: 用户桌面应用 (Electron)
  选项B: 用户VPS部署 (Docker容器)
  选项C: 混合部署 (云节点 + 用户节点)

技术要求:
  - 容器化部署 (Docker)
  - 自动更新机制
  - 健康检查接口
  - 资源限制配置
```

## 扩展性设计

### 水平扩展策略
```yaml
无状态服务:
  - API网关: 基于负载均衡器扩展
  - 业务服务: Kubernetes自动扩缩容
  - 执行节点: 按用户需求动态增加

有状态服务:
  - 数据库: 读写分离，分库分表
  - 缓存: Redis集群
  - 消息队列: RabbitMQ集群
```

### 容量规划
```yaml
初期 (100用户):
  - 数据库: 10GB存储
  - 缓存: 2GB内存
  - 带宽: 100Mbps
  - 执行节点: 100个容器

中期 (1000用户):
  - 数据库: 100GB存储，读写分离
  - 缓存: 16GB内存，Redis集群
  - 带宽: 1Gbps
  - 执行节点: 1000个容器

长期 (10000用户):
  - 数据库: 分库分表，多区域部署
  - 缓存: 多区域Redis集群
  - 带宽: 10Gbps，CDN加速
  - 执行节点: 按需自动扩缩容
```

## 监控架构

### 监控层次
```yaml
基础设施层:
  - 服务器: CPU、内存、磁盘、网络
  - 容器: 资源使用、重启次数
  - 网络: 延迟、丢包、带宽

应用层:
  - 服务: 响应时间、错误率、吞吐量
  - 数据库: 查询性能、连接数、锁等待
  - 缓存: 命中率、内存使用、延迟

业务层:
  - 用户行为: 活跃用户、功能使用
  - 任务执行: 成功率、平均时长、失败类型
  - 账号健康: 登录成功率、封号率
```

### 告警策略
```yaml
紧急告警 (P0):
  - 服务不可用
  - 数据库连接失败
  - 大量任务失败

重要告警 (P1):
  - 响应时间超过阈值
  - 错误率上升
  - 资源使用率过高

警告告警 (P2):
  - 单个任务失败
  - 缓存命中率下降
  - 磁盘空间不足

信息通知 (P3):
  - 每日执行报告
  - 系统维护通知
  - 新功能发布
```
