# 8. 部署架构

## 部署环境规划

### 环境分类
```yaml
开发环境 (Development):
  - 用途: 开发人员本地开发和测试
  - 部署: Docker Compose
  - 数据: 测试数据，可随时重置
  - 访问: 仅内网访问

测试环境 (Staging):
  - 用途: 集成测试和预发布验证
  - 部署: Kubernetes集群
  - 数据: 模拟生产数据
  - 访问: 受限外部访问

生产环境 (Production):
  - 用途: 线上用户使用
  - 部署: 多区域Kubernetes集群
  - 数据: 真实用户数据
  - 访问: 公开访问
```

### 基础设施需求
```yaml
计算资源:
  - API服务器: 4核8GB × 3节点
  - 数据库: 8核16GB × 2节点（主从）
  - 缓存: 4核8GB × 3节点（Redis集群）
  - 文件存储: 独立对象存储

网络资源:
  - 负载均衡器: 1个
  - CDN: 全球分发
  - VPN网关: 用于管理访问

存储资源:
  - 数据库存储: 500GB SSD
  - 文件存储: 1TB对象存储
  - 备份存储: 2TB（保留30天）
```

## 容器化部署

### Docker镜像配置
```dockerfile
# 后端API Dockerfile
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
RUN npm run build

FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./package.json
EXPOSE 3000
CMD ["node", "dist/main.js"]

# 前端PWA Dockerfile
FROM node:18-alpine AS frontend-builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:alpine
COPY --from=frontend-builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/nginx.conf
EXPOSE 80
```

      - postgres

  # 前端PWA
  frontend:
    build: ./frontend
    ports:
      - "8080:80"
    depends_on:
      - api

### 部署配置示例
```yaml
# API服务部署
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-deployment
  namespace: fbautobot-production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api
  template:
    metadata:
      labels:
        app: api
    spec:
      containers:
      - name: api
        image: fbautobot/api:1.0.0
        ports:
        - containerPort: 3000
        env:
        - name: NODE_ENV
          value: production
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: connection-string
        resources:
          requests:
            memory: "512Mi"
            cpu: "250m"
          limits:
            memory: "1Gi"
            cpu: "500m"
        livenessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 30
          periodSeconds: 10
        readinessProbe:
          httpGet:
            path: /ready
            port: 3000
          initialDelaySeconds: 5
          periodSeconds: 5
---
# API服务暴露
apiVersion: v1
kind: Service
metadata:
  name: api-service
  namespace: fbautobot-production
spec:
  selector:
    app: api
  ports:
  - port: 80
    targetPort: 3000
  type: ClusterIP
---
# Ingress路由
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
  namespace: fbautobot-production
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  rules:
  - host: api.fbautobot.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80
```

## 网络架构

### 网络拓扑
```
用户 → CDN → 负载均衡器 → Kubernetes Ingress → 服务网格 → Pod
                                     ↓
                             外部服务（数据库、缓存、存储）
```

### 安全组规则
```yaml
入站规则:
  - 80/tcp: HTTP流量（CDN -> 负载均衡器）
  - 443/tcp: HTTPS流量（CDN -> 负载均衡器）
  - 22/tcp: SSH管理（仅允许管理IP）

出站规则:
  - 所有协议: 允许所有出站流量
```

### 域名和SSL
```yaml
域名配置:
  - api.fbautobot.com: API服务
  - app.fbautobot.com: 前端PWA
  - monitor.fbautobot.com: 监控面板

SSL证书:
  - 使用Let's Encrypt自动签发
  - 通配符证书: *.fbautobot.com
  - 自动续期
```

## 存储架构

### 数据存储策略
```yaml
关系数据:
  - 存储: PostgreSQL（主从复制）
  - 备份: 每日全量 + 每小时增量
  - 保留: 30天

缓存数据:
  - 存储: Redis集群（3主3从）
  - 持久化: AOF + RDB
  - 内存: 16GB每节点

文件存储:
  - 存储: MinIO集群（4节点）
  - 类型: 对象存储（S3兼容）
  - 容量: 1TB初始，自动扩展

日志存储:
  - 存储: Loki + S3
  - 保留: 30天热数据，180天冷数据
```

### 备份策略
```yaml
数据库备份:
  - 频率: 每日02:00全量备份
  - 增量: 每小时WAL日志备份
  - 存储: 独立备份服务器 + 云存储
  - 恢复测试: 每周执行恢复测试

文件备份:
  - 频率: 每日04:00增量备份
  - 存储: 跨区域复制
  - 版本: 保留7个版本

配置备份:
  - 频率: 实时（Git版本控制）
  - 存储: GitHub私有仓库
  - 加密: 敏感配置加密存储
```

## 监控和告警

### 监控栈
```yaml
指标收集:
  - Prometheus: 指标收集和存储
  - Node Exporter: 节点指标
  - kube-state-metrics: Kubernetes指标

可视化:
  - Grafana: 仪表板和可视化
  - 预置仪表板:
    - 系统资源使用
    - 应用性能指标
    - 业务指标监控

日志收集:
  - Loki: 日志聚合
  - Fluent Bit: 日志收集和转发
  - 日志级别: INFO（生产），DEBUG（开发）

追踪:
  - Jaeger: 分布式追踪
  - 采样率: 10%（生产），100%（开发）
```

### 告警规则
```yaml
紧急告警 (P0):
  - 条件: 服务不可用超过5分钟
  - 通知: 短信 + 电话 + 应用内
  - 响应: 立即处理

重要告警 (P1):
  - 条件: 错误率 > 5% 持续10分钟
  - 通知: 应用内 + 邮件
  - 响应: 2小时内处理

警告告警 (P2):
  - 条件: 资源使用率 > 80%
  - 通知: 应用内
  - 响应: 24小时内处理

信息通知 (P3):
  - 条件: 每日执行报告
  - 通知: 邮件
  - 响应: 无需立即处理
```

## 安全架构

### 网络安全
```yaml
网络隔离:
  - VPC划分: 生产、测试、开发环境隔离
  - 安全组: 最小权限原则
  - 网络策略: Kubernetes NetworkPolicy

传输安全:
  - TLS 1.3: 所有外部通信
  - 证书管理: 自动续期和轮换
  - 密钥交换: 前向保密

访问控制:
  - IAM角色: 基于角色的访问控制
  - 多因素认证: 管理访问
  - API密钥: 服务间通信
```

### 数据安全
```yaml
加密:
  - 传输中: TLS加密
  - 静态数据: AES-256加密
  - 密钥管理: HSM或KMS

访问控制:
  - 数据库: 最小权限账户
  - 存储: 基于策略的访问控制
  - 审计: 所有数据访问记录

合规:
  - 数据分类: 敏感数据标识
  - 保留策略: 符合法规要求
  - 用户权利: 数据访问和删除权
```

## 高可用设计

### 多区域部署
```yaml
主区域 (ap-southeast-1):
  - 位置: 新加坡
  - 角色: 主区域，全功能部署
  - 数据库: 主库
  - 用户: 亚太地区用户

备用区域 (us-east-1):
  - 位置: 弗吉尼亚
  - 角色: 灾难恢复
  - 数据库: 从库（异步复制）
  - 用户: 美洲地区用户（只读）

故障转移:
  - 检测: 健康检查每30秒
  - 切换: 自动（数据库），手动（应用）
  - RTO: < 15分钟（应用），< 1小时（数据库）
  - RPO: < 5分钟
```

### 自动扩缩容
```yaml
水平扩缩容 (HPA):
  - API服务: CPU > 70% 或内存 > 80%
  - 副本数: 最小2，最大10
  - 冷却时间: 扩容3分钟，缩容5分钟

垂直扩缩容 (VPA):
  - 数据库: 基于查询性能
  - 缓存: 基于内存使用
  - 存储: 基于容量使用

成本优化:
  - 定时扩缩: 根据使用模式调整
  - 竞价实例: 非关键工作负载
  - 资源预留: 保证核心服务资源
```

## 持续部署流程

### CI/CD流水线
```yaml
开发阶段:
  - 触发: Git Push到feature分支
  - 步骤: 代码检查 → 单元测试 → 构建镜像
  - 环境: 部署到开发环境
  - 验证: 自动化测试

测试阶段:
  - 触发: Merge到develop分支
  - 步骤: 集成测试 → E2E测试 → 安全扫描
  - 环境: 部署到测试环境
  - 验证: 人工验收测试

生产阶段:
  - 触发: Merge到main分支
  - 步骤: 性能测试 → 蓝绿部署 → 监控验证
  - 环境: 部署到生产环境
  - 验证: 金丝雀发布 + A/B测试
```

### 部署策略
```yaml
蓝绿部署:
  - 策略: 同时运行新旧版本
  - 流量切换: 一次性切换
  - 回滚: 快速切换回旧版本
  - 适用: 主要版本更新

金丝雀发布:
  - 策略: 逐步增加新版本流量
  - 流量比例: 1% → 5% → 25% → 50% → 100%
  - 监控: 关键指标监控
  - 适用: 日常功能更新

滚动更新:
  - 策略: 逐个Pod更新
  - 最大不可用: 1个Pod
  - 最大激增: 1个Pod
  - 适用: 配置更新和小修复
```

## 灾难恢复

### 恢复目标
```yaml
恢复时间目标 (RTO):
  - 应用服务: < 15分钟
  - 数据库: < 1小时
  - 完整系统: < 4小时

恢复点目标 (RPO):
  - 用户数据: < 5分钟
  - 配置数据: < 1分钟
  - 日志数据: < 15分钟
```

### 恢复流程
```yaml
检测阶段:
  - 监控系统检测故障
  - 自动告警触发
  - 故障分类和评估

响应阶段:
  - 启动应急响应团队
  - 评估影响范围
  - 决定恢复策略

恢复阶段:
  - 切换到备用区域
  - 恢复数据库和存储
  - 验证系统功能

复盘阶段:
  - 分析根本原因
  - 改进预防措施
  - 更新应急预案
```
